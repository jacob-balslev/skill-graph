# Skill Graph Metadata Contract

## Design Principles

This contract is the public source of truth for the Skill Graph frontmatter format.

It does:
- keep a flat author-facing frontmatter format
- keep one `SKILL.md` file per skill
- keep one generated manifest downstream
- tighten field semantics
- add a small number of high-value fields beyond the Agent Skills base
- stay additive to the Agent Skills standard so every Skill Graph skill can be transformed back to the base shape

Tooling to consume this contract is planned — see `docs/plans/scripts-roadmap.md`.

## Relationship to the Agent Skills standard

Skill Graph is a graph-aware superset of the [Agent Skills](https://agentskills.io/specification) open standard. The base standard requires two frontmatter fields (`name` and `description`) and defines four optional fields (`license`, `compatibility`, `metadata`, `allowed-tools`). Skill Graph keeps the two required base fields. It keeps three of the four optional base fields (`license`, `compatibility`, `allowed-tools`) as top-level Skill Graph fields. It does not use the base `metadata` field; Skill Graph promotes its own extensions to additional top-level fields instead of nesting them under `metadata`.

| Field | Source | Skill Graph treatment |
|---|---|---|
| `name` | Agent Skills required | Kept as required; Skill Graph tightens the character pattern |
| `description` | Agent Skills required | Kept as required; scoped to routing |
| `license` | Agent Skills optional | Kept top-level; strongly recommended for shared skills |
| `compatibility` | Agent Skills optional | Kept top-level; optional |
| `allowed-tools` | Agent Skills optional | Kept top-level as a space-separated string |
| `metadata` | Agent Skills optional | Not used at the top level; Skill Graph promotes extensions to named fields |
| `schema_version`, `version`, `type`, `family`, `scope`, `owner`, `freshness`, `drift_check`, `eval_status` | Skill Graph extension | Required for Skill Graph; additive to the base |
| `relations`, `domain_frame`, `portability`, `triggers`, `keywords`, `paths`, `route_groups`, `extends`, `stability` | Skill Graph extension | Optional in Skill Graph; additive to the base |

A Skill Graph `SKILL.md` is **not** a valid Agent Skills file as authored, because Skill Graph requires fields the base standard does not define. An export transform can produce an Agent-Skills-valid file by moving every Skill Graph extension field under the standard `metadata:` key. The top-level fields in the exported file are then only `name`, `description`, `license`, `compatibility`, `allowed-tools`, and `metadata`. Tooling for this transform is on the roadmap as `scripts/export-skill.js`. Until that ships, the `agent-skills` value in `portability.exports` describes a compatibility goal, not a working export path.

## Contract Model

There are two artifacts:

1. `SKILL.md`
   - source of truth authored by humans
2. `skills.manifest.json`
   - compiled normalized inventory generated by tooling

The source contract should stay readable.
The manifest can be richer and more normalized.

## Example Template Rule

The example template in this repo is intentionally concrete.

- It is a real specimen of valid end-state usage, not a placeholder scaffold.
- It should contain real-looking, internally consistent values.
- It should never teach authors or agents to ship fake values such as `your-skill-name`, `path/to/file`, or other placeholder sludge.
- The teaching layer lives in structurally distinct out-of-band slots, not in hollow placeholder content and not in cargo-cult-able H2 headers.

### Semantic layer discipline

Two fields in the contract look similar but serve opposite layers. They are sibling layers of progressive disclosure, not duplicates:

| Element | Layer | Purpose | Length |
|---|---|---|---|
| `description:` (frontmatter) | **Routing contract** | Tells the router "should this skill activate?" — pushy trigger phrases plus an explicit negative boundary | ≤ 3 sentences |
| `## Coverage` (body section) | **Scope map** | Tells the reader, once the skill is loaded, what topics the skill covers in detail | Bulleted topic list, ≥ 4 items for non-trivial skills |

If `description:` is bloated with scope detail, it will conflict with `## Coverage`. If `## Coverage` restates the description in one line, it has collapsed into the routing layer. Restore each to its proper layer when this happens.

### Teaching layer delivery mechanisms

Meta-commentary aimed at the template reader must never live in an H2 header slot. AI agents adapting a template copy its H2 structure verbatim. That means meta sections like `## How To Read This Template` or `## Omitted On Purpose` get cargo-culted into every new skill. The correct delivery mechanisms are:

- **`> **TEMPLATE NOTE:**` blockquotes** for body-level meta guidance (visible on GitHub, structurally distinct from H2 headers in the markdown AST)
- **`# TEMPLATE NOTE:` inline YAML comments** in the frontmatter for field-level meta guidance (disappears naturally when an author tightens a new skill's frontmatter)
- **Never** put meta-commentary in an H2/H3 section header — those belong to the skill's canonical sections only

### When adapting the example

1. Restore `description:` to routing-only if it has drifted into scope description.
2. Keep the H2 structure that matches the skill archetype (see **Archetype section map** below).
3. Replace example values only with equally real, context-correct values.
4. Remove sections that are conditionally irrelevant rather than keeping them and filling with fake content.
5. Leave `> **TEMPLATE NOTE:**` blockquotes and `# TEMPLATE NOTE:` YAML comments out of the new skill — they are authoring scaffolding, not skill content.

### Archetype section map

Each skill archetype expects a specific set of body H2 sections. These are the minimum required sections per archetype. Additional sections are allowed when they earn their line count.

| Archetype | Required H2 sections |
|---|---|
| `capability` | `## Coverage`, `## Philosophy`, `## Verification`, `## Do NOT Use When` |
| `workflow` | `## Coverage`, `## Philosophy`, `## Workflow`, `## Verification`, `## Do NOT Use When` |
| `router` | `## Coverage`, `## Routing Rules`, `## Do NOT Use When` |
| `overlay` | `## Coverage`, `## Overlay Rules`, `## Extends` (name of the base skill), `## Do NOT Use When` |

`## Key Files` is recommended for skills that reference concrete repo files. `## References` is recommended for skills that point at external reading.

The goal is to teach agents what a finished skill should look like in practice. The delivery mechanism must not pollute the final artifact.

## Field Groups

These groups are documentation categories only. They do not require nested YAML.

### Identity

```yaml
schema_version: 1
name: shopify

version: 1.0.0

```

### Classification

```yaml

```

### Activation

```yaml
triggers:
  - shopify-skill
keywords:
  - shopify
  - shopify api
  - shopify webhook

  - src/integrations/shopify/**
```

```yaml
relations:
  adjacent:
    - webhook-integration
  boundary:
    - fulfillment
  verify_with:
    - test-coverage
  depends_on:
    - api-key-management
```

```yaml
domain_frame:
  domain_object: Shopify integration behavior
  evaluation_mode: repo_specific

    - src/integrations/shopify/webhooks.ts

    - webhook_signature_bypass

### Governance

  - integrations
```

## Conditional Requiredness

The flat optional list is not enough on its own. Some fields are optional globally but required for specific skill classes.

### Required for all skills

```yaml
schema_version
name
description
version
type
family
scope
owner
freshness
drift_check
eval_status
```

### Required for routable skills

Use this group when the skill is expected to activate from user language or label-driven routing.

```yaml
keywords
```

`triggers` and `paths` are optional overall, but should be treated as required when the routing design explicitly depends on labels or file-surface activation.

### Required for overlay skills

```yaml
extends
```

### Required for grounded skills

Use this group when the skill makes concrete repo or implementation claims.

```yaml
domain_frame
```

### Optional enrichments

These improve portability or discoverability, but are not required for a valid v1 skill.

```yaml
compatibility
allowed-tools
route_groups
portability
```

`stability` and `license` are listed under **Strongly Recommended Fields** above — every non-trivial skill should set them, even though the schema does not enforce them.

## Field Semantics

### `schema_version`

Purpose:
- version the contract explicitly
- allow future migration tooling

Rule:
- integer or stringified integer accepted
- start at `1`

### `name`

Purpose:
- stable skill identifier used for routing, relations, and filesystem layout

Rule:
- must match the regular expression `^[a-z0-9][a-z0-9-/:]*$`
- lowercase alphanumerics, hyphens, forward slashes, and colons only
- must start with a letter or digit
- should match the parent directory name for Agent-Skills-compatible export

Note: The Agent Skills base standard uses a stricter name pattern (no `/`, no `:`, no consecutive hyphens, no trailing hyphen). A Skill Graph name that uses `/` or `:` can still be exported to Agent Skills by normalizing the separators in the transform layer.

### `type`

Purpose:
- defines behavioral archetype only

Allowed values:
- `capability`
- `workflow`
- `router`
- `overlay`

Rule:
- keep `type` restricted to the four allowed archetype values
- `family` is the field for browse taxonomy and domain grouping, not `type`

### `family`

Purpose:
- human browse taxonomy only

Rule:
- helps discovery and grouping
- does not imply runtime behavior or evaluation logic

Recommended usage:
- `family` is open-ended in the v1 OSS contract
- use stable, human-readable buckets
- avoid one-off synonyms for the same idea

Examples:
- `knowledge`
- `engineering`
- `frontend`
- `quality`
- `integration`

If a future release adopts a controlled family vocabulary, that change should be versioned explicitly.

### `scope`

Purpose:
- indicates locality and usage mode at a high level

Allowed values:
- `operational`
- `reference`
- `generic`

Rule:
- remove `overlay` from `scope`
- use `type: overlay` and `extends` for overlay semantics

### `extends`

Purpose:
- explicitly states which base skill an overlay extends

Rule:
- only valid when `type: overlay`

### `relations`

Purpose:
- graph semantics between skills

Rules:
- `adjacent`: discoverability and related reading
- `boundary`: anti-ownership and wrong-skill routing protection
- `verify_with`: verification or co-loading candidates
- `depends_on`: explicit dependency where ordering or conceptual reliance matters

### `domain_frame`

Purpose:
- declares what the skill governs in the real world or codebase
- provides evidence anchors for repo-grounded verification

Use when:
- skill is repo-specific
- skill makes concrete implementation claims
- skill is validated against code or docs, not just abstract reasoning

### `stability`

Purpose:
- tells users how mature the skill is

Allowed values:
- `experimental`
- `stable`
- `frozen`
- `deprecated`

### `eval_status`

Purpose:
- makes audit and eval readiness visible in authored metadata

Allowed values:

| Value | Meaning | Expected artifact state |
|---|---|---|
| `none` | no eval work started | no eval artifacts expected |
| `pending` | eval work is intended but not authored yet | example or future eval work may exist elsewhere |
| `evals` | eval artifacts exist | one or more eval files should be present |
| `evals+trigger` | eval artifacts exist and trigger coverage is being checked | eval files plus trigger-aware review |
| `passing` | eval artifacts exist and are currently passing | eval files plus successful verification |
| `active` | actively maintained and evaluated in a live toolchain | eval files plus active maintenance loop |

Rule:
- use `pending` only when missing eval artifacts are an intentional temporary state
- use `evals` or higher once concrete eval files ship with the skill

### `portability`

Purpose:
- declares which agent runtimes a skill can be exported to

Rule:
- single structured object with `level` and `exports`
- `level` must be one of `high`, `medium`, `low`
- `exports` is an array constrained to the enum in `schemas/skill.schema.json`

### `allowed-tools`

Purpose:
- names the pre-approved tools a skill is permitted to invoke when loaded into an agent runtime that honors the declaration

Format:
- space-separated string (matches the [Agent Skills](https://agentskills.io/specification) base standard)
- not an array — the string form is a hard requirement for Agent-Skills-compatible export

Example:
```yaml
allowed-tools: Read Grep Bash
```

Scope:
- Experimental per the Agent Skills spec — support varies across agent implementations
- Skill Graph validates the shape but does not enforce the allowlist at runtime

### `compatibility`

Purpose:
- declares environment requirements (intended product, system packages, network access) for skills that have specific runtime needs

Format:
- string, maximum 500 characters (per Agent Skills spec)
- omit entirely when a skill is fully generic with no environment requirements

Example:
```yaml
compatibility: Requires Node.js 18+ and git
```

- `scope`
- `owner`

- `extends`
- `domain_frame`
- `route_groups`
- `license`
- `compatibility`
- `allowed-tools`
- `portability`
- `stability`

### Generated in `skills.manifest.json`

- normalized category rollups
- health flags
- eval counts
- missing coverage flags
- mirror status
- compiled activation tables
- generated docs ownership

## Authored-to-Generated Mapping

This table defines the intended public mapping for downstream tooling.

| Authored field | Generated manifest field | Notes |
|---|---|---|
| `name` | `id`, `name` | `id` may be normalized from `name`; `name` remains human-readable |
| `description` | `description` | copied through |
| `version` | `version` | copied through |
| `type` | `type` | copied through |
| `family` | `family` | copied through |
| `scope` | `scope` | copied through |
| `owner` | `owner` | copied through |
| `extends` | `extends` | copied through when present |
| `keywords`, `triggers`, `paths` | `activation.*` | grouped under `activation` |
| `relations.*` | `relations.*` | grouped under `relations` |
| `domain_frame.*` | `grounding.*` | normalized under `grounding` |
| `portability.*` | `portability.*` | copied through when present |
| `eval_status`, `freshness`, `drift_check` | `health.*` | grouped under `health` |

Generator-only fields such as summary rollups, path normalization, mirror status, and health flags belong only in `skills.manifest.json`.

## Schema Strictness

The v1 OSS schemas are intentionally strict.

- unknown top-level fields should fail validation rather than being silently accepted
- field names should not rely on undocumented aliases
- new public fields should be added by updating both the docs and the schemas

## Non-Goals

This contract does not introduce:
- open-core vs closed-core runtime layers
- nested metadata requirements for authoring
- enterprise-only fields
- a second skill format
- a marketplace-specific packaging system

It also does not require a full private control plane. The OSS contract keeps only the metadata needed for a portable, graph-aware, lintable system.
