# Changelog

All notable changes to Skill Graph are recorded here.

The format is based on [Keep a Changelog 1.1.0](https://keepachangelog.com/en/1.1.0/) and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

Skill Graph versions describe the contract at each checkpoint in the git history and are how consumers pin. Beginning with 0.5.0, the repository is package-ready as `@skill-graph/cli` with the `skill-graph` binary. `schema_version` in the JSON Schemas tracks contract shape (currently `8` — owned by `@skill-graph/cli` post-consolidation; see [ADR 0009](docs/adr/0009-sibling-repo-deprecation.md) for the consolidation addendum and the post-2026-05-18 source-of-truth model, [ADR 0011](docs/adr/0011-split-audit-verdict-into-four-verdicts.md) for the v6→v7 four-verdict Audit Status split, and [ADR 0017](docs/adr/0017-five-axis-classification-model.md) for the v7→v8 classification model. The dual-versioning rationale historically lived in `skill-metadata-protocol/docs/adr/0007-version-source-of-truth.md`, which is preserved in the now docs-only mirror at `jacob-balslev/skill-metadata-protocol` for inbound-link stability — that mirror is **frozen / read-only** per ADR-0009 § Update and is referenced here for historical context only, not as an active ADR series entry); this changelog tracks the library as a whole (tooling + starters + docs).

## [Unreleased]

### Added

- **Heartbeat + active-Monitor wrapper for long audit-loop batches (2026-05-29).** Long graded runs (`evaluate`, `audit --graded`, `evolve --top N`) were being run as blind background tasks, which are silent through a crash or hang — indistinguishable from "still running," the recurring operator failure mode. Added `scripts/watch-audit-batch.sh`: a heartbeat-driven active listener that emits one event per `PROGRESS` / `HANG` (a unit past `--cell-stall`) / `STALE` (heartbeat stopped ticking) / `COMPLETE` (terminal), covering every terminal state so silence never reads as success. Pairs with a `status.json` heartbeat contract (`{ts,pid,total,done,failed,running:[{cell,elapsed_s}],complete}`) the batch runner writes on every unit start/finish + a ≤15s tick; operators stream it via the Claude Code `Monitor` tool. Documented in `SKILL_AUDIT_LOOP.md` § "Running long batches in the background." Reference producer of the heartbeat is the A/B experiment driver (`dist/ab/comprehension-ab-driver.js::writeStatus()`); wiring the heartbeat into the canonical `evaluate`/`evolve`/`batch-eval` runners is a tracked follow-up (until then, `--proc <pattern>` gives liveness-based death/stall detection).
- **`skill-graph render` — first-class local compile of the library into consumable skills (2026-05-29).** The marketplace export is only the public-publish profile; `render` is the primary deployment path for ACTUAL USE — yours locally or any cloner's. `scripts/render-skills.js` (CLI: `skill-graph render`, npm: `render` / `render:check`) compiles EVERY canonical skill — including `deployment_target: project` skills the marketplace publication gate excludes — into plain Agent-Skills `<name>/SKILL.md` files at `--out` (default `dist/skills`, git-ignored), each carrying clean frontmatter + the authored body + the projected `## Skill Graph context` section. No publication gate, no privacy scrub, no skills.sh URL rewriting (it is a local artifact, not a publish). `--profile full` (default, every meaningful field) or `--profile runtime` (behavioral fields only — omits lifecycle/audit/provenance); `--check` for CI staleness. Point `--out` at a runtime's skills dir (e.g. `~/.claude/skills`) to consume directly. The body-projection renderer was extracted to the shared `scripts/lib/render-skill-context.js` (with stable `<!-- skill-graph-context:start/end -->` fence markers) so `render` and `export` share one compile core; `export-marketplace-skills.js` now imports it. Coverage in `scripts/__tests__/test-render-skills.js`. Docs: README § Consuming the best-quality skills locally. Verified (all exit 0): render test, marketplace-export test, export-parser-drift, `npm run verify:system`.
- **Export-time body projection — readable `## Skill Graph context` section (2026-05-29).** Vendor auto-loaders (Claude/Codex/OpenCode/Gemini) read `name` + `description` at startup and the SKILL.md body on activation, but never the exported `metadata:` map — so the graph's classification, relations, Understanding fields, grounding, lifecycle, and audit status were invisible to consuming agents even though present in the file. `scripts/export-marketplace-skills.js::renderSkillGraphContext()` now projects every meaningful Skill Metadata Protocol field into a generated, deterministic `## Skill Graph context` section appended to each exported skill body (via a new `bodySuffix` hook on `scripts/export-skill.js::buildExportedSkill()`, which leaves generic single-skill exports byte-identical when unused). The `metadata:` map is preserved unchanged for lossless round-trip; the description boundary projection (2026-05-26) is untouched; pure machine-state fields (`urn`, drift hashes, `eval_failed_ids`, telemetry) stay in the map only. Slugs render as `code` spans (no Markdown links → no dangling-link findings), output is deterministic so `--check` stays stable, and the rendered strings already passed the privacy gate via the `metadata:` map so the change is privacy-neutral. All 154 marketplace exports regenerated. Coverage in `scripts/__tests__/test-marketplace-export.js`. Docs: `docs/marketplace-syndication.md` § Export-Time Body Projection. Verified (all exit 0): marketplace-export test, export-parser-drift, `export --check`, `marketplace:verify` (154/154), `verify-skill-md-export` (156/156), `test:unit`.

The package `skill-improvement-helpers.js` was already the portability-improved superset (no change needed). The coupled grader prompts were unified into `lib/audit/graders/` — the package concept grader absorbed the workspace's "Anti-Compression Mandate" + "Calibration Reference Set" block — and `grader-output-validator.js` moved into `lib/audit/`. ### Changed

- **v7 → v8 clean cut (2026-05-27).** Per [AGENTS.md § Major Version Is a Clean Cut](AGENTS.md), the live tree describes v8 only. The cut consolidated the iterative v8 design — initial classification model 2026-05-26 (per [ADR-0017](docs/adr/0017-five-axis-classification-model.md), closing SH-6547); `operation` axis retirement 2026-05-27 commit `f88603d`; and entity-vocabulary disambiguation + `workspace_tags` removal 2026-05-27 — into the final v8 contract documented below.

  **Required classification axes:** `subject` (closed 9-enum browse shelf) + `deployment_target` (closed 2-enum `portable`/`project`; the `workspace` enum value removed).

  **Required free-text axis:** `scope` — PRD-style label, repurposed from the prior closed enum. Value phrasing standardized.

  **Optional structural axes:** `subjects[]` (polyhierarchy, max 2, primary first), `taxonomy_domain` (slash-delimited sub-path, renamed from `domain`), `project[]` (array of `{handle, role}` belonging-entity references), `repo[]` (array of `{handle, url}` references). The four structural axes `deployment_target` / `project[]` / `repo[]` / `grounding` are orthogonal — no field's value is derived from another.

  **Companion doctrinal renames (commit `f88603d`):** `description` is now an about-statement (not a routing contract); "Eval-health triple" renamed to "Evaluation Status"; "Health Block" renamed to "Audit Status"; "Domain affiliation" → "Projects"; `scope` value phrasing standardized. Audit Status preserves the four discrete verdicts (`structural_verdict` / `truth_verdict` / `comprehension_verdict` / `application_verdict`); `application_verdict` is the primary quality signal.

  **Grounding rename:** `grounding.domain_object` → `grounding.subject_matter`; the v3.1 `grounding.subject` alias retired (`scripts/lib/alias-contract.js` drops the alias pair).

  **Removed from schema:** `type`, `archetype`, `category`, `categories`, `secondary_categories`, `primaryCategory`, `layerPrimary`, `routingRole`, `family`, `layer`, `operation`, `eval_status`, `workspace_tags`. The 6 category-based `allOf` branches removed. The `type:overlay → extends` and `type ≠ overlay → no extends` `allOf` branches removed. The `codebase` and `reference` scope-enum values removed. Manifest aggregates `summary.by_type` / `summary.by_category` / `summary.by_operation` and the `skills[].operation` field removed. ## [0.3.0] — 2026-04-17

The "contract honesty + graded audits" release. Three priorities from the internal-only OSS roadmap (`docs/plans/skill-graph-oss-roadmap.md`): narrow the `portability.targets` enum to what actually has a working transform (P1), promote the audit runner from a lint-stub seeder to a real prompt-driven grader (P2), and close the versioning-policy-to-reality gap with pinned v2 schema files (P3). No GitHub remote, no npm publish — this release matures the contract for internal use.

### Added
- `scripts/lib/audit-prompt-builder.js` — 7-dimension registry, context collector (reads `SKILL.md` plus `grounding.truth_sources`), IDENTITY-STEPS-OUTPUT prompt composer, response parser (requires a `<verdict>…</verdict>` JSON block), and coarse verdict aggregator. Powers the new graded audit mode.
- `scripts/skill-audit.js --graded` mode — calls an explicit external grader CLI once per scorecard dimension and merges the responses into `findings.md` / `verdict.md` / `scorecard.md`, replacing TODO placeholders with evidence-backed PASS / PASS WITH FIXES / FAIL verdicts. Grader CLI is user-supplied via `--grader-cli`; no API key is embedded and no default provider is selected by the script.
- `scripts/lib/mock-grader.js` — deterministic stand-in grader that produces canned `<verdict>` blocks per dimension. Lets CI smoke-test the graded pipeline without an API key and powers the `examples/audits/documentation/` graded fixture.
- `schemas/skill.v2.schema.json` and `schemas/manifest.v2.schema.json` — pinned v2 copies of the unversioned schemas, content-identical modulo `$id` and `title`. Consumers that want stability across a future v3 bump pin to these; consumers that want to follow latest use the unversioned files.
- `check-contract-consistency.js` C6 — enforces parity between the versioned and unversioned schemas. Drift is an error; the v2 files must track the unversioned files exactly until v3 ships.
- `CHANGELOG.md` (this file) — Keep-a-Changelog-formatted history seeded with the current bundle. Future releases get entries under `## [Unreleased]` as they land.

### Changed
- `examples/audits/documentation/` regenerated via the mock grader to demonstrate what `--graded` output looks like: real scores, `N/A` for grounding on a portable skill, evidence-cited findings.
- `docs/integrations/github-actions.md` — installation guidance rewritten so the primary path for that release was clone-and-vendor. The then-future `npm install --save-dev skill-graph` / `npx skill-graph-lint` snippets were preserved as a secondary **WHEN PUBLISHED** section so no reader mistook them for current 0.3.0 guidance.
- `docs/skill-metadata-protocol.md § Schema Versioning Policy` — rewrites policy point 3 as a current-state description now that the versioned schema files exist.
- `README.md` Status section — audit runner moves from "stub generator" bullet to a two-mode description and out of the Planned list. Validation table expands to six checks (adds C6). Quick Tour lists unversioned and v2 schemas side by side.
- `docs/plans/scripts-roadmap.md § Audit runner` — marked SHIPPED with full two-mode description, grader-CLI discipline, and links to the new `scripts/lib/audit-prompt-builder.js` and `scripts/lib/mock-grader.js`.

### Removed (breaking)
- `portability.targets` enum values `cursor`, `windsurf`, `copilot`, and `agents-md`. Only `skill-md` has a working transform (`scripts/export-skill.js`); the other four were compatibility goals that violated the contract's `additionalProperties: false` strictness rule. The enum now accepts only `["skill-md"]`. Re-adding a runtime is gated on a new RFC and the same PR that ships its transform. Starter skills, the template, the sample manifest, the exports fixtures, and all docs that mentioned the removed values were updated to match.
